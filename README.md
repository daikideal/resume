## 株式会社ビットキー プラットフォーム開発部 (2023/10 ~ 現在)

### SAML/SCIM連携作業の自動化 (2025/04 ~ 2025/05)

#### 概要

従来、開発者が手作業で行なっていたSCIM/SAML連携作業をAPI経由で完結できる機能を設計・実装した。

#### 担当

チームメンバー:

- エンジニア: 3名

役割:

- 業務シーケンスの整理・図の作成
- API設計の叩き台作成

#### 使用技術

- Go
- Google Cloud
  - Identity Platform
  - Cloud Firestore
- HTML
- JavaScript

#### 課題

前提として、自社プロダクトをSP(Service Provider)とした SAML SSO・SCIMプロビジョニング の機能はすでに認証認可基盤チームにより開発・運用されていた。運用される中で2つの課題が出てきた。

1. 開発当初は連携作業依頼はそれほど件数も多くなく特に困っていなかったが、営業の方針としてエンタープライズ向けの導入が増え、連携作業依頼が増加してきていた
2. また、対応しているIdPの一つであるOkta様より、OIN(Okta Integration Network)に弊社プロダクトを載せないかという旨の提案があった。連携手順をOINに掲載することになるが、手作業のままだと「弊社窓口まで連絡」という手順を載せることになり、開発者がボトルネックになってスケールしない懸念があった

#### 取り組み

- SAML/SCIM連携の導入と運用をお客様の手で完結できるようにするための機能をリリース
  - Identity Platform Admin SDKを使用したSAMLプロバイダー作成API
  - SAML証明書更新API
  - SCIMシークレットの発行API
  - SCIMシークレット更新API
- API経由での連携作業をプロダクト側の開発者が体験できるよう、HTMLとJavaScriptで動くデモサイトを社内検証環境にリリース

#### 工夫した点

プロジェクト始動当初はSAML/SCIM連携の**導入**だけが視野に入っていて、"SAMLプロバイダー作成API"と"SCIMシークレットの発行API"だけだったが、業務シーケンスの整理にあたってSAML/SCIM連携についてのユースケースを洗い出していると、過去に「証明書の期限が切れていたのでSSOに失敗した」という問い合わせがお客様より寄せられていたことを発見した。
その問い合わせは開発者が手作業でお客様のIdPのメタデータXMLより証明書の値を取得してIdentity Platformに入力することで対応したが、証明書の更新もAPI経由でできたほうがよりリードタイムを減らし、お客様に自走していただけるプロダクトになると考え、このタイミングでAPIとして実装することにした。

#### 成果

認証認可基盤側の機能としては、SAML/SCIM連携を手作業なくAPI経由のみで完結させられるようになった。
2025年11月にプロダクト側でSAML/SCIM連携機能の実装が開始したため、プロダクトを通した市場投入の時期は未定。

### Cloud SQLのサービス無停止アップグレード (2025/04 ~ 2025/05)

#### 概要

Cloud SQL for PostgreSQLをサービス無停止でアップグレード

#### 担当

チームメンバー:

- エンジニア: 3名

役割:

- アップグレード方法の選定
- アップグレード作業

#### 使用技術

- Cloud SQL Enterpriseエディション
- PostgreSQL

#### 課題

家庭向け・オフィス向けスマートロックサービス共通の認証認可基盤のデータベースのアップグレードであるため、メンテナンス期間を設けることができない。

その他の前提:

- プライマリインスタンス×1、レプリカインスタンス×3
- アプリケーションはマイクロサービスで、データベースは3つに分かれている
- アプリケーションはGKEクラスタで稼働
- コネクションプールは使用していない

#### 取り組み

Postgres組み込みの論理レプリケーション機能を使用して旧DBから新DBへデータを移行し、アプリケーションからの接続を切り替える作業を実施

#### 工夫した点

3つの方法を検討した。

1. インプレースアップグレード
2. Google CloudのCloud DMS
3. Postgres組み込みの論理レプリケーション

1.は、プライマリをアップグレードするためにレプリカを停止する必要があり、プライマリも20分~1時間停止する。レプリカからアップグレードしてフェイルオーバーも考えたが、Postgresはバージョンが異なるインスタンス同士のレプリケーションをサポートしない。どうしてもサービス停止期間が発生するため不採用。

2.は、データベースごとに移行元インスタンスをプライマリ・移行先インスタンスをレプリカとして"接続プロファイル"を作成することでデータを同期することになる。しかしDMSの仕様上、移行元インスタンスから移行先インスタンスに対して同時に複数の"接続プロファイル"を作成することはできず、かつ移行先インスタンスのプロモート時にはデータ不整合を防ぐためにアプリケーションを停止することがGoogle Cloud公式より推奨されている。よって、作業期間が長引くうえにサービス停止期間が数回発生することになるため不採用。

3.は、自前でコマンドを実行して移行元・移行先インスタンス間のレプリケーションを組む手間がかかるものの、複数のデータベースを同時にデータ同期することができるので最終的な作業期間は長くはならない。さらに、レプリケーションを組んだあとはアプリケーション側のデータベース接続情報を移行元から移行先へと修正してローリングアップデートすればデータ不整合などは発生しない見込みが立ったため、採用。

検証環境で検証し、データ不整合やその他のトラブルも発生しないことを確認したので本番環境で決行したところ、503エラーが一時的に数件発生したのみで、問題なくアップグレードが完了した。

#### 成果

Cloud SQL for PostgreSQLを、PostgreSQL 14からPostgreSQL 16へメンテナンス期間等を設けずにアップグレードすることができた。

学びとして以下のような知見を得た。

- PostgreSQLは、無停止でのメジャーバージョンアップに役立つ論理レプリケーションという機能が組み込みで存在する
- マイクロサービスごとにデータベースを分ける場合、特定のサービスのアップグレードが他のサービスに影響を与えないようにインスタンスごと分ける方がよい

### 顔認証用ライブラリを後方互換性を維持してアップデート (2024/09 ~ 2024/11)

#### 概要

認証のために顔画像から顔特徴量を抽出するライブラリ(FaceMe SDK)のアップデート計画をVPoPや他チームと調整し、自ら設計・実装も行った。

#### 担当

チームメンバー:

- エンジニア3名

役割:

- VPoPや他チーム、SDKベンダーとのコミュニケーション
- SDKの動作検証
- バックエンドの設計・実装

#### 使用技術

- Go
- Google Cloud
  - Pub/Sub
- C++

#### 課題

前提として、FaceMe SDKによる顔特徴量抽出機能は従来より市場投入されていた。
認証認可基盤側でマスターとなる顔画像と顔特徴量を登録し、クライアントとなるプロダクト側では端末に顔をかざすとその場で特徴量を抽出し、認証認可基盤で持つ特徴量とアプリが抽出した特徴量の類似度で認証を行う。
この顔特徴量には"モデル"という概念が存在し、新しいモデルほど精度の高い認証を実現できる。

SDKのベンダーよりSDKの最新バージョンがリリースされた旨の連絡をいただいた。アップデート内容としては、最新の顔特徴量モデルが利用できるようになったほか、顔特徴量の抽出速度や特徴量同士の比較速度が上がるとのことだった。
プロダクト側のロードマップとして、最新のSDKにバージョンアップして顔認証のUXを向上させるという計画があったが、認証認可基盤側が対応しなければプロダクト側でも最新モデルが使用できないため、プロジェクトとしてSDKのアップグレードを行うことを決定した。

最新バージョンのSDKを組み込んだコンテナをビルドして動作検証を行ったところ、市場で現在も使用している古いモデルが廃止されていることが判明。
古いSDKを搭載したアプリを利用している顧客は多く、その端末はビルのエレベーターホールやオフィスの入り口に設置されているケースもあり、顧客との調整が必要で、すぐの完全移行ができない。愚直に認証認可基盤側でSDKのアップグレードを行ってリリースしてしまうと、顔情報の登録を行えないという事象が発生する恐れがあった。

#### 取り組み

- 新旧SDKを同時に稼働させる設計を行い、最新モデルへ移行するスケジュールも含めてVPoPやプロダクト開発チームと合意
- C++で実装されたSDK、およびSDKとの繋ぎ込み機能のデバッグ
- Goを使用したバックエンド開発

#### 工夫した点

認証認可基盤はマイクロサービスで、大きくは以下４つに分かれている。

- REST APIサービス
- DBとのアクセスを担うサービス
- 顔画像・顔特徴量の保存を担うサービス
- 顔特徴量抽出サービス (SDKを搭載)

顔情報登録のユースケースは、"REST APIサービス"でプロダクト側から顔画像とモデルの指定を含んだリクエストを受けて、"顔画像・顔特徴量の保存を担うサービス"へ繋ぎ、"顔画像・顔特徴量の保存を担うサービス"から"顔特徴量抽出サービス"へPub/Subで顔特徴量抽出依頼のメッセージを送ることで実現している。

顔特徴量抽出サービスに新旧SDKを両方組み込み、入力値であるモデルの指定に応じて呼び出すSDKを分岐することも考えたが、SDKは1台のマシンにつき1つしかアクティベートできず、SDK1つにつきライセンス契約が必要という制限も発覚した。
そこで、新旧バージョンのSDKをそれぞれ別のコンテナで実行し、古いモデルでの顔情報登録は旧バージョンのSDKを搭載したコンテナで、新しいモデルでの顔情報登録は新バージョンのSDKを搭載したコンテナで処理するアプローチを考えた。
この設計をプロダクト側とVPoPに提案、旧モデル廃止・新モデル移行にかかる期間を見積もっていただき、その期間ぶんだけライセンスを購入、プロダクト側でも移行計画を実施するという内容で合意した。

実装については、メッセージを新旧コンテナでどう振り分けるかで悩んだ。

1. 既存のPub/Subトピックを旧コンテナ用とし、新しく新コンテナ用のPub/Subトピックを追加。"顔画像・顔特徴量の保存を担うサービス"からはモデルの指定に応じてpublishするトピックを分岐させる
2. 既存のPub/Subトピックはそのままに、新コンテナ用のサブスクリプションを追加。サブスクリプションフィルターを活用して新旧コンテナ各々が対応するモデルでの顔特徴量抽出依頼のメッセージのみを購読する

旧コンテナはプロダクトが最新モデルの利用へ完全移行でき次第破棄する計画だったので、その際にトピックごと破棄すればいいだけ、かつ既存のインフラコンポーネントに変更を加えなくてもよい案1.を採用した。

#### 成果

認証認可基盤としては機能的な後方互換を保ったまま、SDKのアップグレードを完遂した。
2025年10月には、プロダクト側からの古いモデルの利用も完全になくなり、認証認可基盤側でも古いモデルを廃止できた。

### パスキー認証機能の実装 (2023/10 ~ 2023/12)

#### 概要

自社の複数プロダクトを支える認証認可基盤にパスキー認証機能を実装

#### 担当

チームメンバー:

- エンジニア: 3名

役割:

- 処理シーケンス図の作成
- プロトタイプの実装
- バックエンド・フロントエンドの実装

#### 使用技術

- Go
- JavaScript (Web Authentication API)
- Firebase Hosting

#### 課題

プロダクトへのログイン方法はID・パスワードによる認証がメインだが、パスワードはフィッシングの被害を受けやすく安全とは言えない。
UXを損ねずにパスワード以上の安全性を実現できるパスキー認証機能を認証認可基盤に実装し、プロダクトに提供することで顧客の安全で快適なサービス利用に貢献できると考えられた。

#### 取り組み

- 選定したライブラリ(go-webauthn/webauthn)を使用して、パスキー認証のUXを体験できるプロトタイプを作成し、チームに共有
- Goを使用したバックエンドの開発
- HTML/JavaScriptを使用したパスキー認証のテストページの開発。テストページはFirebase Hostingでデプロイし、Datadogのブラウザテストでパスキー登録・認証ができるかを日次で確認
- VPoP・プロダクト側のマネージャー・デザイナーをお呼びし、テストページを使って、パスキーの必要性・パスキー認証のUXについて説明するデモを実施

#### 工夫した点

認証基盤とプロダクトでドメインが異なることによりパスキー認証機能が提供できない問題に対し、リクエストから動的に RP ID を設定するアプローチを立案・実装し、解決した。

詳細はこちらのスライドで解説: https://speakerdeck.com/bitkey/challenges-and-solutions-for-passkey-authentication-implementation-in-authentication-and-authorization-infrastructure-supporting-multiple-brands-b71cf3bc-925d-441d-9c01-337f1bd26c73

　
デモにおいては、技術的な知識の非対称性に配慮して平易な言葉を選んだり、実際にパスキー登録~認証を体験してもらうことで、パスワード認証と比べてのメリットや、パスキーのUXについてイメージを持ってもらいやすくした。

#### 成果

認証認可基盤の機能としてはパスキー認証を備えるに至った。

学びとして以下のような知見を得た。

- 複数の異なるブランド(ドメイン)のプロダクトを展開する企業の認証認可基盤にパスキーを実装する際は、RP IDの管理に工夫が必要である

### 入退室管理SaaSの速度改善 (2023/01 ~ 2023/06)

#### 概要

自社開発toB SaaSのページの初期描画までの速度改善

#### 担当

チームメンバー:

- PdM: 1名
- エンジニア: 4名

役割:

- ボトルネックの調査
- 修正方針の設計・実装

#### 使用技術

- TypeScript
- Node.js
- React
- Cloud Firestore

#### 課題

不具合による顧客からの問い合わせが多く、チームとして新規機能の開発を止めて品質の改善を行う意思決定がなされた。

中でも温度感の高いものとして、「サイトの挙動が重い」という類のものがあった。
「ページを開くのにx秒ほどかかってしまう」というものから、顧客が利用するPCのスペックによってはブラウザがクラッシュしてしまいそもそも機能が利用できないという事象も確認されていた。

#### 取り組み

リファインメント・プランニングにおいて優先的に取り組むと決めた機能から順に、顧客がどういう操作をするときの・どの処理が遅いのかボトルネックを調査し、解消していった。

#### 工夫した点

ボトルネックの特定は、基本的に以下の手順で行った:

1. ローカル環境に状況を再現
2. React Developer Toolsで表示の遅いコンポーネントを特定
3. コンポーネント内で行っている処理に`console.time()`を追加して計測

4つの機能・画面の速度改善を担当。

- 予約履歴詳細画面
  原因は、予約履歴に付随して画面に表示しないといけないデータを取得するために ①Firestoreにクエリ ②メモリ上でfilter、という処理を行なっていたが、①のwhere条件ではあまり結果を絞り込めておらず、大量の不要なデータを毎回クエリしていたことだった。
  ②のfilterの条件と同等のwhere条件でFirestoreへのクエリを行ってから、①のwhere条件と同等のfilter処理を行うように修正し、最終的な絞り込みの条件を変えずに速度だけを改善した。Firestoreのインデックスに手を加えても良かったが、当時はFirestoreからRDBに移行しようという別プロジェクトがあったため、Firestoreに変更を加えずに実現できる方法を採用した。
- アクセスコントロール詳細画面
  アクセスコントロールとは、いつ・誰が・どのスペースに のデータを統合して保存しており、実際に入退室を制御するリソースのことである。
  原因は、アクセスコントロールのとあるパラメータにユニーク制約があり、詳細画面でアクセスコントロールの設定を編集するときにこのパラメータの重複チェックを行うため、画面描画時に大量のデータをフェッチしてメモリ上に保持していたことだった。
  重複チェックのバリデーション処理をアクセスコントロール編集APIの方に移すことで解決した。
- 従業員詳細画面
  従業員に紐づいたアクセスコントロールも表示するという仕様になっている。
  原因は、クエリするアクセスコントロールの量が多すぎること。
  取り扱えるデータ量を減らせないかとアクセスコントロールの仕様を確認したところ、スペースの利用予約を行うと都度アクセスコントロールが作成され、作成されたアクセスコントロールはその予約でしか参照されないことを理解した。BigQueryで本番データを確認したところ、アクセスコントロール515,000件のうち97%が予約起因であることが判明。「予約起因のアクセスコントロールと同等の情報は予約履歴画面で確認できるので、従業員詳細画面で取り扱うアクセスコントロールに予約起因のものは含めなくてもよいのではないか」と考え、PdMと相談したところ承諾されたので、クエリの絞り込み条件から予約起因のアクセスコントロールを除外して解決。
- スペース詳細画面
  スペースに紐づいたアクセスコントロールも表示するという仕様になっている。
  従業員詳細画面と類似の原因・対応方針となったので割愛。

いま振り返った時の反省点として、ボトルネックの特定はRUMを使うとより効率的だったかもしれないと思う。

#### 成果

修正前の初期描画速度と比較して、約75~95%の速度改善を実現:

- 予約履歴詳細画面 75%改善 (10sec → 2.5sec)
- アクセスコントロール詳細画面 90%改善(50sec → 3sec)
- 従業員詳細画面 96%改善(55sec → 2sec)
- スペース詳細画面 90% ~ 97%改善 (1min~3min → 5sec)
